## 总体结论

这个插件的功能骨架比较完整，已经覆盖语法高亮、补全、Hover、定义跳转、文档符号和实时诊断。`src/validator.test.js` 也沉淀了较多控制流、注释、中文字符和括号匹配测试，这是后续持续修正新代宏程序规则的好基础。

当前主要风险不在“功能有没有”，而在几处实现和数据源之间已经出现漂移：语法高亮、补全、诊断、snippet、生成脚本和说明文档对同一条宏程序规则的理解并不完全一致。建议优先修复会直接影响编辑体验或产生误判的点，再建立一套发布前检查清单。

## 高优先级问题

### 1. TextMate 语法把普通圆括号当注释

位置：`syntaxes/syntec-macro.tmLanguage.json`、`build/build_grammar.py`

当前语法规则包含：

```json
{
  "begin": "\\(",
  "end": "\\)",
  "name": "comment.block.paren.syntec-macro"
}
```

这会把表达式里的普通括号也高亮成注释，例如 `ABS(#1 - #2)`、`IF (#1 = 1) THEN`。从手册看，ISO 格式中 `( ... )` 可作为注释，但 MACRO 格式注释是 `(* ... *)`，而插件语言本身是 syntec-macro。建议移除该规则，或仅在确实需要兼容 ISO 文件时通过单独语言/模式处理。

### 2. 补全触发条件过窄

位置：`src/extension.js`

`provideCompletionItems` 只有在 `prefix === 'G' || prefix === 'G6'` 时补全 G 代码，只有在 `prefix === 'M'` 时补全 M 代码。用户输入 `G0`、`G01`、`G66`、`M0`、`M98` 时不会继续得到候选项。

建议改为按前缀过滤：

- `prefix.startsWith('G')` 时从 `keywords.gcodes` 中按 `startsWith(prefix)` 过滤。
- `prefix.startsWith('M')` 时从 `keywords.mcodes` 中按 `startsWith(prefix)` 过滤。
- 对 `G66.1` 这类带点 G 码，需要确认 VS Code word range 和触发字符是否能覆盖 `.`，必要时增加触发字符或专门处理。

### 3. 配置项没有实际生效

位置：`package.json`、`src/extension.js`

`package.json` 暴露了 `syntecMacro.enableDiagnostics`、`syntecMacro.enableCompletions`、`syntecMacro.enableHover`，但代码中没有读取这些配置。用户关闭这些开关后，诊断、补全和 Hover 仍会继续工作。

建议：

- 在 provider 和 `refreshDiagnostics` 中读取 `vscode.workspace.getConfiguration('syntecMacro')`。
- 监听 `workspace.onDidChangeConfiguration`，配置变化后刷新诊断或让 provider 返回空结果。
- 注意配置属性当前放在 `syntecMacro` 下，但属性名写成了 `enableDiagnostics`，实际键通常会显示为 `syntecMacro.enableDiagnostics`。实现时要统一读取方式。

### 4. GOTO / N 标签规则需统一

位置：`src/validator.js`、`src/extension.js`、`syntaxes/syntec-macro.tmLanguage.json`、`snippets/syntec-macro.json`、`README.md`

README 和更新日志强调控制器实测规则为 `GOTO 100;`，目标 `N100;`。当前诊断收集 N 标签时确实只接受分号结尾，但其他地方仍比较宽松或不完全一致：

- `provideDefinition` 用 `l.startsWith(targetLabel)`，会把 `N1000;` 误当成 `GOTO 100` 的目标。
- `provideDocumentSymbol` 用 `/^N(\d+)/`，会把 `N100`、`N100:`、`N100 ABC` 都加入大纲。
- TextMate 高亮用 `\bN\d+\b`，没有体现 `N数字;` 的严格格式。
- `extractGotoTarget` 接受 `GOTO 100`，没有要求分号；这可以作为宽容诊断，但建议明确是“兼容识别”还是“严格合法”。

建议建立一个共享的标签解析函数，并让诊断、跳转、大纲和高亮尽量使用同一规则。若要兼容旧写法，可区分“识别目标”和“报告格式错误”。

### 5. 生成脚本输出目录错误且内容落后

位置：`build/build_grammar.py`、`build/build_snippets.py`

两个脚本使用 `os.path.dirname(__file__)` 作为基准，当前会输出到：

- `build/syntaxes/syntec-macro.tmLanguage.json`
- `build/snippets/syntec-macro.json`

但扩展实际贡献的是根目录下：

- `syntaxes/syntec-macro.tmLanguage.json`
- `snippets/syntec-macro.json`

此外，生成脚本内容落后于当前产物，例如 `build/build_grammar.py` 仍包含 `INT`，snippet 脚本中的 `if` / `ife` 前缀、`REPEAT` 结束形式、`N10;` 等也和现有 `snippets/syntec-macro.json` 不一致。

建议尽快修正脚本输出路径，并把当前已校准的 grammar/snippet 反向同步回脚本，否则后续执行 `npm run build` 会生成无效目录或旧规则产物。

## 中优先级建议

### 6. 宏程序文件跳转只查工作区根目录

位置：`src/extension.js`

`findMacroFile` 只用 `fs.existsSync` 和 `fs.readdirSync` 查工作区根目录。仓库中样例宏程序集中在 `macro-samples/`，实际项目也常按机台、客户、功能目录组织宏文件，因此 `G65 Pxxx` 跳转容易找不到目标。

建议改用 `vscode.workspace.findFiles`，支持递归查找和忽略目录。可以增加配置项，例如 `syntecMacro.macroSearchPaths`，默认搜索整个工作区或常见目录。

### 7. Hover 对变量基本不可达

位置：`src/extension.js`

`provideHover` 的 word range 使用 `/[A-Za-z_][A-Za-z0-9_]*/`，这不会把 `#1`、`#ABC`、`@100` 作为完整 word 捕获。因此后面的变量 Hover 分支 `if (/^[#@]/.test(word))` 大多不会触发。

建议使用与 `language-configuration.json` 中 `wordPattern` 接近的范围规则，或在 Hover 中根据当前位置向左右手动扩展 `#` / `@` 变量范围。

### 8. 关键字清单分散，容易继续漂移

位置：`src/keywords.js`、`src/validator.js`、`syntaxes/syntec-macro.tmLanguage.json`、`build/build_grammar.py`

替代结束关键字如 `ENDIF`、`ENDFOR`、`ENDWHILE`、`ENDCASE`、`ENDREPEAT` 在 validator 和 grammar 中有支持，但 `src/keywords.js` 里的补全清单没有完全包含。函数清单也存在 `src/functions.js` 比 grammar 更完整的情况，例如字符串和位运算函数在补全里有，但部分高亮分类未覆盖。

建议建立单一数据源：

- `src/functions.js` 负责补全/Hover，同时可生成 grammar 的函数匹配。
- `src/keywords.js` 负责关键字、替代关键字、G/M 码，同时可被 validator 和 grammar 复用。
- build 脚本只从这些 JS/JSON 数据生成 TextMate grammar 和 snippets，避免手写多份列表。

### 9. 诊断规则对 `EXIT` 的语义可能过强

位置：`src/validator.js`、`src/validator.test.js`

当前逻辑遇到 `EXIT` 会从栈里移除最近循环块，导致后续 `END_FOR` / `END_WHILE` 被报“没有匹配”。如果新代宏程序的 `EXIT` 是运行期跳出循环，而源码仍必须保留结构性 `END_FOR`，那么这里会误报。测试文件目前把“EXIT 后不应有 ENDFOR”写成预期，但这条规则建议再次用控制器实测确认。

建议将 `EXIT` 处理改为：只检查是否位于循环内，默认不改变结构栈；如果控制器确实要求 `EXIT` 后不能再写结束关键字，再保留当前行为并在 README 明确说明。

### 10. VS Code configurationDefaults 里有非标准键

位置：`package.json`

`editor.comment`、`editor.blockCommentStart`、`editor.blockCommentEnd` 不是 VS Code 标准配置项。注释样式已经在 `syntaxes/language-configuration.json` 中定义，建议删除这些无效默认项，避免误导维护者。

### 11. 版本号显示不一致

位置：`src/extension.js`

状态栏显示 `Syntec Macro v1.4.0`，但 console log 仍输出 `扩展已激活 v1.3.7`。建议统一从 `package.json` 读取版本，避免发布时漏改。

## 低优先级建议

### 12. 增加扩展级测试

目前测试集中在 `validateDocument`，但补全、Hover、Definition 和 DocumentSymbol 没有自动化验证。建议添加最小 VS Code extension test 或把纯逻辑拆出后用 Node 测试覆盖：

- 输入 `G0` 仍能补全 `G00` / `G01`。
- `GOTO 100;` 只跳到 `N100;`，不跳到 `N1000;`。
- `#1`、`@100` 能显示变量 Hover。
- 配置关闭诊断后不产生 diagnostics。

### 13. 补全项可进一步优化插入体验

位置：`src/extension.js`

函数补全当前只插入函数名，不带括号和参数占位。变量补全通过 `#` 后触发时插入 `1` 是合理的，但 completion label 是 `#1`，建议确认 VS Code 中实际替换范围是否稳定。函数可考虑使用 `SnippetString`，例如 `ABS(${1:num})`。

### 14. 增加发布前检查清单

建议在文档中记录每次发布前至少执行：

- `npm run build`，确认生成文件写入正确目录且没有旧规则回退。
- `node src/validator.test.js`，确认诊断测试通过。
- 用 VS Code Extension Development Host 手动验证高亮、补全、Hover、跳转、配置开关。
- 检查 `package.json` 版本、状态栏版本、CHANGELOG 版本一致。
- 打包前检查 `.vscodeignore`，确保必要文档和语法文件进入 VSIX。
